StormSpace Agent Framework

Orchestrator Pipeline

An autonomous, multi-agent development lifecycle — from specification through implementation, review, regression testing, and knowledge capture.

Architecture Overview

The Orchestrator coordinates a linear pipeline of 11 stages. Each stage invokes specialised sub-agents, tracks progress in a persistent progress.md file, and reports status to GitHub via the Delivery Manager. The pipeline halts on any non-Delivery-Manager failure; GitHub tracking is non-blocking.

Agent Invocation Flow

Pipeline Stages

The pipeline progresses linearly through 11 stages. Each stage has defined inputs, outputs, and gate conditions. The Delivery Manager is invoked at every stage transition but is non-blocking.

Stage 1
Initialisation
Creates the task branch (task/{slug}), initialises the progress.md tracking file, and invokes the Delivery Manager to create the GitHub tracking issue with the agent-task label.

GitHub tracking issue #{number}

Specification
The Business Analyst drafts a structured spec.md from the user's request. When --refine is set, it invokes the Question Relay to post clarifying questions on the GitHub issue and poll for user replies before finalising.

Outputs
.agent-context/tasks/{slug}/spec.md
⊘ --refine skips Q&A

Stage 3
Planning
The Planner analyses the spec, launches parallel Researcher sub-agents to gather codebase context and external documentation, then produces a master plan and per-phase detail files. If research reveals ambiguities, it invokes the Question Relay for user clarification before finalising. Each phase maps to one future commit.
Planner Question Relay Delivery Manager
Outputs
.agent-context/tasks/{slug}/plan.md
.agent-context/tasks/{slug}/phase-{N}-{name}.md
GitHub phase sub-issues

Stage 4
Plan Review
The Refiner sends the plan to the Adversarial Reviewer (Tribunal) for 1 round of multi-model critique. Three reviewer agents — Claude Opus 4.6, GPT-5.4, and GPT-5.3-Codex — review in parallel and produce a consolidated verdict. The Refiner fixes any critical or major findings.
Refiner Adversarial Reviewer Delivery Manager
Outputs
Refined plan files (fixes applied)
Tribunal verdict
5
Stage 5
Plan Approval
If --auto was set, this stage is skipped. Otherwise, the plan summary is presented to the user for manual approval before any code is written.
Orchestrator Delivery Manager
⊘ --auto skips approval
6
Stage 6
Per-Phase Implementation
A pre-implementation baseline commit is created. Then for each phase (1 through N), the Implementer executes the phase plan file, builds and tests, and the Orchestrator commits the result. The Delivery Manager closes each phase sub-issue after completion.
Implementer Delivery Manager
Outputs
git commit per phase
Phase sub-issues closed
7
Stage 7
Implementation Review
The Implementation Reviewer captures the full diff from the baseline commit, invokes the Adversarial Reviewer tribunal, fixes critical/major findings, and verifies the build and tests pass. This is a hard gate — FAIL halts the pipeline.
Implementation Reviewer Adversarial Reviewer Delivery Manager
✓ PASS → push + create PR ✗ FAIL → pipeline halts
8
Stage 8
Regression Testing
The application is started via the VS Code task. The Regression Tester navigates through user journeys in a real browser using Playwright, prioritising flows affected by the changed files. It reports issues but never fixes them.
Regression Tester Delivery Manager
✓ PASS → knowledge update ✗ FAIL → fix cycle
9
Stage 9
Regression Fix Cycle
If regressions were found, the Implementer is invoked to fix them (max 1 cycle). The Regression Tester re-verifies. If issues persist after one fix attempt, the pipeline halts and reports remaining regressions.
Implementer Regression Tester Delivery Manager
✓ PASS → continue ✗ FAIL → pipeline halts (max 1 cycle)
10
Stage 10
Knowledge Update
The Knowledge Keeper updates .agent-context/knowledge/ documentation to reflect the verified implementation changes. This always runs last, after all implementation and testing is confirmed passing.
Knowledge Keeper Delivery Manager
Outputs
Updated knowledge/*.md files
11
Stage 11
Completion
Final commit and push. The Delivery Manager closes the tracking issue, updates the GitHub project board, and marks the PR ready for merge. The progress file is updated to "Completed" status.
Orchestrator Delivery Manager
Outputs
Final git push
GitHub issue closed

Agent Communication Contracts

Agents communicate through structured output headings. The Orchestrator parses these to gate pipeline flow — no free-form interpretation of results.

Agent Output Contract Values Pipeline Action
Implementation Reviewer ## Review Status: PASS FAIL PASS → push + PR. FAIL → halt pipeline.
Regression Tester ## Regression Status: PASS FAIL PASS → knowledge update. FAIL → fix cycle.
Delivery Manager ## Delivery: {CMD} INIT, SPEC_COMPLETE, PLAN_READY, PHASE_END, ... Extract IDs (issue #, node IDs). Non-blocking on error.
Business Analyst ## Spec: COMPLETE — Verify spec.md was written. Proceed to planning.

Key Concepts

Progress Tracking
progress.md
Persistent state file created at pipeline start, updated after every stage. Records timestamps, GitHub IDs, phase statuses, and an issues log. All GitHub tracking IDs are passed from this file when invoking the Delivery Manager.
Git Strategy
Commit per Phase
A baseline commit is created before phase 1. Each phase gets its own commit. The Implementation Reviewer diffs from baseline to HEAD for full-scope review. The branch is pushed only after review passes.
Quality Gate
Multi-Model Tribunal
Three AI models (Opus 4.6, GPT-5.4, GPT-5.3-Codex) review in parallel. Each brings a different perspective — findings are consolidated and severity-ranked. Used for both plan review and code review.
Safety Model
Non-Blocking DM
The Delivery Manager (GitHub tracking) is non-blocking. If it fails on init, a flag is set and all subsequent DM calls are skipped. Pipeline progress is never blocked by GitHub API issues.
Automation Flag
--auto Mode
Detected in the user message at pipeline start. Skips the manual plan approval gate (Stage 5), allowing fully autonomous execution from spec to completion without human intervention.
Refinement Flag
--refine Mode
When set, the Business Analyst invokes the Question Relay to post clarifying questions on the GitHub issue and poll for user replies. Enables interactive specification refinement before planning begins.
Async User Input
Question Relay
A reusable utility agent (Haiku 4.5) that posts numbered questions to a GitHub issue and polls for user replies at configurable intervals. Used by the Business Analyst (--refine) and the Planner (alignment phase) to gather clarification without blocking the pipeline indefinitely.

Error Handling

The pipeline distinguishes between fatal failures (which halt execution) and non-blocking warnings (which are logged but allow continuation).

Fatal — Halts Pipeline
Any Non-DM Agent Failure
If any agent other than the Delivery Manager fails, the error is logged to the progress file issues log, the task status is set to "Failed", the Delivery Manager is notified with complete (status: failed), and the pipeline halts immediately. Partial changes remain in the working tree — no rollback is attempted.
Fatal — Halts Pipeline
Implementation Review FAIL
If the Implementation Reviewer returns ## Review Status: FAIL, the pipeline halts. No code is pushed, no PR is created. The tracking issue is marked as failed with a summary of review findings.
Fatal — Halts Pipeline
Regression Persists After Fix
Maximum 1 regression fix cycle. If regressions persist after one Implementer fix attempt, the app is stopped, the pipeline halts, and remaining regressions are reported to the user. No further loops.
Warning — Continues
Delivery Manager Failure
Delivery Manager errors are logged to the issues log but never halt the pipeline. If the init command fails, a github-tracking-disabled flag is set and all subsequent DM calls are skipped silently.

Task File Structure

Every pipeline run produces a set of files under .agent-context/tasks/{slug}/. These files are committed to the task branch and serve as a full audit trail. 

.agent-context/
├── tasks/
│   └── {task-slug}/
│       ├── progress.md        ← Pipeline state, timestamps, GitHub IDs
│       ├── spec.md            ← Structured specification (Business Analyst)
│       ├── plan.md            ← Master plan with phase overview (Planner)
│       ├── phase-1-{name}.md  ← Phase 1 detailed implementation steps
│       ├── phase-2-{name}.md  ← Phase 2 detailed implementation steps
│       └── phase-N-{name}.md  ← Phase N ...
└── knowledge/
    └── *.md               ← Updated by Knowledge Keeper (Stage 10)
